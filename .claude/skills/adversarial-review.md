# Adversarial Review

> 核心：同一模型写的代码再让它自己审，容易陷入相同思维盲区。不同模型交叉审查才能覆盖盲区。

## 触发条件

- 变更涉及核心业务逻辑
- 变更文件数 >= 5 或行数 >= 200
- io-wy 说「对抗审查」「adversarial review」「cross-model review」「多模型审查」

## 三大盲区

| 盲区 | 表现 | 根因 |
|------|------|------|
| **知识盲区** | 某个框架/模式理解不完整 | 训练数据差异 |
| **注意力盲区** | 大 diff 下后半部分审查质量下降 | 上下文窗口有限 |
| **确认偏差** | 发现一个问题后沿同一方向深挖 | 认知惯性 |

## 执行流程

### Step 1：收集变更

```bash
git diff master...HEAD
```

### Step 2：主模型审查

审查维度：

| 维度 | 检查点 |
|------|--------|
| 正确性 | 逻辑正确、边界条件、错误路径 |
| 安全性 | 注入、鉴权绕过、密钥泄露、SSRF |
| 并发 | 共享状态保护、锁粒度、死锁 |
| 资源 | goroutine 泄漏、连接泄漏、句柄泄漏 |
| 性能 | 热路径开销、N+1、不必要分配 |
| 可观测 | 错误有日志、关键路径有 trace |

每个问题标注：级别（Critical/Major/Minor/Suggestion）+ 文件:行号 + 分类 + 修复建议。

输出问题集 A。

### Step 3：交叉模型审查

调用 `/codex-review` 独立审查，输出问题集 B。

### Step 4：交叉验证

| 情况 | 处理 |
|------|------|
| A 交 B 都发现 | 高置信度，确认问题 |
| 仅 A 发现 | 标注 `[主模型独有-需确认]` |
| 仅 B 发现 | 标注 `[交叉模型独有-需确认]` |

### Step 5：输出合并报告

```markdown
# 对抗式审查报告

## 概要
| 级别 | 确认 | 需确认 |
|------|------|--------|
| Critical | x | x |
| Major | x | x |
| Minor | x | x |

## 确认问题（高置信度）
### C1. [分类] <标题>
- 文件: file.go:123
- 发现者: 主模型 + codex
- 问题: ...
- 建议: ...

## 需确认问题
### M3. [分类] <标题> [主模型独有]
- 文件: file.go:45
- 问题: ...
```

## 审查区分能力

- **本次引入** vs **历史遗留**（只审本次引入的）
- **风格偏好** vs **实际缺陷**（只提实际缺陷）
- **正确代码被误标** vs **真正错误**（不确定标 Suggestion）

## 禁止行为

- 禁止同一模型写+审（至少用不同模型）
- 禁止把历史遗留混入本次审查
- 禁止把风格偏好标为 Critical/Major
