# Competitive Analysis

> 核心：实现新功能前先看业界怎么做。不是照抄，是站在前人肩膀上做更好的决策。

## 触发条件

- 设计新的 API 接口或协议
- 选择技术方案时有 >= 2 个候选项
- 实现非 trivial 的业务逻辑（如认证、限流、缓存策略）
- io-wy 说「竞品分析」「调研」「compare」「competitive analysis」「benchmark」

## 执行流程

### Phase 1：确定调研范围

在开始调研前，和 io-wy 确认以下信息：

1. **调研目标**：要解决什么问题？（如"设计 LLM API 的流式响应协议"）
2. **调研维度**：关注哪些方面？（如性能、兼容性、易用性、扩展性）
3. **竞品范围**：哪些产品/项目是参考对象？（如 3-5 个开源项目或商业产品）

输出调研计划，等待 io-wy 确认。

### Phase 2：多子代理并行调研

每个竞品分配一个子代理（Agent tool），并行执行：

```
子代理 A：调研竞品 1
  - WebSearch 搜索竞品 1 的设计文档 / API 文档
  - DeepWiki 阅读竞品 1 的源码（如果是开源项目）
  - 提取关键设计决策、数据结构、接口定义

子代理 B：调研竞品 2
  - 同上流程

子代理 C：调研行业标准和最佳实践
  - WebSearch 搜索 RFC / 规范文档
  - 搜索社区讨论、博客、技术评测
```

每个子代理的输出格式：

```markdown
## [竞品名称] 调研报告

### 基本信息
- 项目/产品名：
- 版本：
- 技术栈：
- Star 数 / 用户量：

### 核心设计
- 整体架构：
- 关键数据结构：
- API 设计：

### 调研维度分析
| 维度 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| 性能 | ... | ... | ... |
| 兼容性 | ... | ... | ... |
| 易用性 | ... | ... | ... |

### 可借鉴的点
1. ...

### 需要避免的坑
1. ...
```

**子代理执行规范**：
- 输出必须 Write 到 `tmp/competitive-[name].md`
- 返回主脑的摘要 <= 200 字，含文件路径
- 每个子代理上下文 <= 8KB

### Phase 3：对比分析与推荐

主脑汇总所有子代理的调研结果，输出对比矩阵：

```markdown
# 竞品对比分析

## 对比矩阵

| 维度 | 竞品 A | 竞品 B | 竞品 C | 行业标准 |
|------|--------|--------|--------|----------|
| 方案 | ... | ... | ... | ... |
| 性能 | ★★★ | ★★ | ★★★★ | ★★★ |
| 复杂度 | 低 | 中 | 高 | 中 |
| 扩展性 | ★★ | ★★★ | ★★★★ | ★★★ |

## 关键发现
1. 所有竞品都采用了 X 模式 → 说明这是行业共识
2. 竞品 B 的 Y 方案有已知问题 → 应避免
3. 行业标准推荐 Z → 值得参考

## 推荐方案
基于以上分析，推荐采用 [方案]，原因：
1. ...
2. ...
3. ...
```

### Phase 4：进入 brainstorming-with-context

调研完成后，带着调研结果进入 `brainstorming-with-context` 的 Phase 3（Design Options），将竞品分析作为方案设计的输入。

## 常见错误

| 错误 | 后果 | 预防 |
|------|------|------|
| 不调研直接实现 | 重复发明轮子，遗漏已知最佳实践 | 触发条件满足时强制执行 |
| 调研过度，分析瘫痪 | 花太多时间在调研上 | 限定 3-5 个竞品，每竞品 <= 30 分钟 |
| 只看优点不看缺点 | 盲目采用竞品方案 | 每个竞品必须列出"需要避免的坑" |
| 照抄竞品实现 | 侵权风险 + 不适配自身架构 | 借鉴思路，不抄代码 |
| 调研结果不落地 | 调研报告写完就忘 | Phase 4 强制衔接 brainstorming |

## 验证清单

- [ ] 调研范围已和 io-wy 确认
- [ ] 至少调研了 2 个竞品
- [ ] 每个竞品覆盖了所有调研维度
- [ ] 对比矩阵已生成
- [ ] 推荐方案有明确的理由
- [ ] 调研结果已作为 brainstorming 输入
